# micro-ROS 与嵌入式协同

## 何时使用 micro-ROS

当 MCU 需要直接成为 ROS 2 计算图中的节点时，优先评估 micro-ROS。

如果 MCU 只是底盘、电机、IMU、编码器或简单传感器控制器，且 Linux 主机负责主要机器人逻辑，那么很多项目里：

- **Linux 侧桥接节点 + 串口/CAN 私有协议**

会比把 MCU 直接接入 DDS 图更简单、更稳、更容易调试。

## 官方架构要点

根据 micro-ROS 官方介绍：

- MCU 侧运行 `Micro XRCE-DDS Client`
- 主机侧运行 `Micro XRCE-DDS Agent`
- micro-ROS 基于 ROS 2 / DDS 生态，用于资源受限设备接入 ROS 2 系统

这意味着：

- 它天然更适合 **ROS 2** 体系
- 如果宿主系统还是纯 ROS 1，通常需要先明确桥接策略，不要强行把 MCU 设计成 ROS 2 节点

## MCU 侧设计原则

- 优先静态内存或可界定的内存使用
- ISR 内不做阻塞、打印、复杂序列化、动态分配
- 严控消息发布频率与执行器负载，避免挤爆串口/以太网链路
- 明确采样时钟、消息时间戳来源和掉线重连策略
- 对底盘/执行器链路实现超时停车、失连保护与看门狗

## 与机器人主机的接口约定

至少明确以下内容：

- 传输介质：UART / CAN / USB / Ethernet
- 帧格式：头、长度、序号、payload、校验、结束符
- 字节序与缩放：角速度、加速度、编码器、姿态、里程单位
- 时间源：MCU 本地 tick、主机时间、同步方式
- 错误处理：重传、丢包、重连、超时、异常复位
- 坐标约定：`base_link`、IMU 安装方向、轮速正方向、里程计积分规则

## 什么时候不建议直接上 micro-ROS

- 主机是纯 ROS 1，短期没有 ROS 2 计划
- MCU 资源非常紧，协议桥已经足够
- 团队更需要可控的串口/CAN 调试链路，而不是 DDS 互操作
- 实时控制环强依赖板端本地闭环，ROS 只做监控与高层命令

## 联调检查单

- 串口/CAN 波特率与缓存是否匹配
- 高速 topic 是否超出链路吞吐
- 时间戳是否来自错误时钟源
- `frame_id` 是否统一
- 板端是否存在动态分配、阻塞队列、无界重试
- 主机侧是否在单线程回调中做了耗时解析
- 是否实现了失连保护、超时保护、复位后的重新握手
