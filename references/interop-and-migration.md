# ROS 1 / ROS 2 互通与迁移

## 适用场景

- 同时存在 ROS 1 与 ROS 2 包
- 计划把部分驱动、算法或上层应用从 ROS 1 迁到 ROS 2
- 需要 `ros1_bridge` 或条件依赖来维持兼容

## 第一原则

- **先分层，再迁移**：把算法、协议解析、设备访问与 ROS API 解耦
- **先桥接，再重构**：先让系统跑起来，再逐步替换节点
- **先统一接口，再迁移实现**：优先稳定消息、服务、动作、TF 与参数语义

## 条件依赖优先于重复分叉

根据 REP-149，`package.xml format="3"` 支持条件依赖。对跨版本共享代码包，可优先考虑：

- 条件化依赖声明
- 将纯算法/工具代码抽成不依赖 ROS API 的公共库
- 将 ROS 1 / ROS 2 的封装层分开维护

这通常比复制两份几乎相同的仓库更可控。

## `ros1_bridge` 使用边界

根据 `ros1_bridge` 官方文档，支持矩阵受 Ubuntu 与 ROS 发行版限制：

- Ubuntu 20.04：支持 ROS 1 Noetic + ROS 2 Humble
- Ubuntu 22.04：ROS 1 官方已不支持，但桥接文档给出 Noetic + Humble/Jazzy 的有限方案
- Ubuntu 24.04：不支持 ROS 1，也不支持 `ros1_bridge`

因此，**不要承诺所有 ROS 1/ROS 2 组合都能直接桥接**。

## 推荐迁移路径

### 路线 A：最稳妥

1. 稳定消息、服务、动作与参数语义
2. 把纯算法与协议层剥离为独立库
3. 先迁边缘节点（工具、可视化、非关键驱动）
4. 用桥接或网关维持系统联通
5. 最后迁核心控制/导航链路

### 路线 B：底层先行

适用于驱动层已有 ROS 2 生态、但上层仍是 ROS 1 的项目：

1. 先迁驱动与传感器节点
2. 通过桥接把数据回送给 ROS 1 上层
3. 再逐步迁定位、导航、任务层

## 编码与构建建议

- 不要在同一个包里混写 `catkin` 与 `ament_*` 构建逻辑，优先拆层或拆包
- 保持每个包只遵循一种构建系统
- 迁移时优先保留 topic 名称、frame、单位和参数名，减少外部联动修改
- 对桥接链路单独做集成验证：发布、订阅、时间戳、频率、TF 都要测

## 回归检查

- topic 是否双向可见
- TF 是否仍然单根且时间连续
- 参数默认值与 launch 覆盖是否保持一致
- 设备驱动在桥接后是否引入明显延迟与丢包
- 导航或控制环的频率、延迟、超时保护是否退化
